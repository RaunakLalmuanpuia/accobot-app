## Phase 1 — Infrastructure

### 1.1 Install Laravel Reverb

```bash
composer require laravel/reverb:^1.4
php artisan reverb:install
```

`reverb:install` publishes `config/reverb.php` and appends Reverb's server config. It does NOT publish a `channels.php` — that must be created manually (see Phase 4).

### 1.2 Install Frontend WebSocket Libraries

```bash
npm install laravel-echo pusher-js
```

These are runtime dependencies (not dev), so they go in `dependencies` in `package.json`.

### 1.3 Install Web Push Library

```bash
composer require minishlink/web-push:^9.0
```

This library requires PHP 8.1+, OpenSSL, and the `gmp` PHP extension. Verify `gmp` is loaded:

```bash
php -m | grep gmp
```

### 1.4 Broadcasting Configuration

**`.env` additions** (append to existing `.env`):

```
# Broadcasting
BROADCAST_CONNECTION=reverb

# Laravel Reverb server settings
REVERB_APP_ID=accobot
REVERB_APP_KEY=your-reverb-key-here
REVERB_APP_SECRET=your-reverb-secret-here
REVERB_HOST=localhost
REVERB_PORT=8080
REVERB_SCHEME=http

# Vite / frontend must know Reverb credentials (public, not secret)
VITE_REVERB_APP_KEY="${REVERB_APP_KEY}"
VITE_REVERB_HOST="${REVERB_HOST}"
VITE_REVERB_PORT="${REVERB_PORT}"
VITE_REVERB_SCHEME="${REVERB_SCHEME}"

# Web Push VAPID keys (generate with: php artisan webpush:vapid)
VAPID_SUBJECT=mailto:admin@yourdomain.com
VAPID_PUBLIC_KEY=
VAPID_PRIVATE_KEY=
VITE_VAPID_PUBLIC_KEY="${VAPID_PUBLIC_KEY}"
```

Generate VAPID keys once and store the output:

```bash
php artisan webpush:vapid
```

**`config/broadcasting.php`** — The file is auto-generated by Laravel 13. After `reverb:install` the `reverb` driver block is added. Verify or add this block manually:

```php
'reverb' => [
    'driver'   => 'reverb',
    'key'      => env('REVERB_APP_KEY'),
    'secret'   => env('REVERB_APP_SECRET'),
    'app_id'   => env('REVERB_APP_ID'),
    'options'  => [
        'host'   => env('REVERB_HOST', '0.0.0.0'),
        'port'   => env('REVERB_PORT', 8080),
        'scheme' => env('REVERB_SCHEME', 'http'),
        'useTLS' => env('REVERB_SCHEME', 'http') === 'https',
    ],
    'client_options' => [],
],
```

Set `'default'` key to `env('BROADCAST_CONNECTION', 'log')` (already in default Laravel config; setting `BROADCAST_CONNECTION=reverb` in `.env` activates it).

### 1.5 Queue Worker

The app already uses the `database` queue driver (confirmed in `config/queue.php`). Broadcasting events are dispatched as queued jobs by default. The existing `php artisan queue:listen` process in the `composer dev` script handles this — no additional configuration needed.

For Reverb's own internal broadcast, Reverb pushes directly over WebSocket (not through the queue). The queue only carries the PHP `BroadcastEvent` wrapper job.

### 1.6 Reverb Server Startup (Development)

Add `php artisan reverb:start` to the `composer dev` concurrently command:

```json
"dev": [
    "Composer\\Config::disableProcessTimeout",
    "npx concurrently -c \"#93c5fd,#c4b5fd,#fb7185,#fdba74,#86efac\" \"php artisan serve\" \"php artisan reverb:start\" \"php artisan queue:listen --tries=1 --timeout=0\" \"php artisan pail --timeout=0\" \"npm run dev\" --names=server,reverb,queue,logs,vite --kill-others"
]
```

## Phase 4 — Laravel Broadcasting

### 4.1 Create `routes/channels.php`

This file does not exist yet. Create it at `routes/channels.php`.

```php
<?php

use App\Models\ChatRoom;
use App\Models\ChatRoomMember;
use Illuminate\Support\Facades\Broadcast;

/**
 * Presence channel for a specific chat room.
 * Channel name: presence-room.{tenantId}.{roomId}
 *
 * Returning a user array joins the presence channel.
 * Returning null/false denies access.
 */
Broadcast::channel('presence-room.{tenantId}.{roomId}', function ($user, string $tenantId, string $roomId) {
    // Verify user belongs to the tenant
    if (! $user->tenants()->where('tenants.id', $tenantId)->exists()) {
        return false;
    }

    // Verify user is a member of this room
    $member = ChatRoomMember::where('chat_room_id', $roomId)
        ->where('user_id', $user->id)
        ->first();

    if (! $member) {
        return false;
    }

    return [
        'id'   => $user->id,
        'name' => $user->name,
    ];
});

/**
 * Private channel for per-user system notifications.
 * Channel name: private-user.{userId}
 */
Broadcast::channel('private-user.{userId}', function ($user, int $userId) {
    return (int) $user->id === $userId;
});

/**
 * Private channel for tenant-wide system notifications (used for the
 * "Notifications" room broadcast so all members receive the event).
 * Channel name: private-tenant.{tenantId}.notifications
 */
Broadcast::channel('private-tenant.{tenantId}.notifications', function ($user, string $tenantId) {
    return $user->tenants()->where('tenants.id', $tenantId)->exists();
});
```

### 4.2 Register `channels.php` in `AppServiceProvider`

In `boot()` add:

```php
require base_path('routes/channels.php');
```

Or, in Laravel 13, add `Broadcast::routes()` and the channels file is auto-discovered if you publish it. The safest approach is explicit inclusion in `AppServiceProvider::boot()`.

## Phase 6 — Laravel Notifications

### 6.1 `SystemNotification` Notification Class

**File**: `app/Notifications/SystemNotification.php`

```
implements ShouldQueue
uses Queueable, SerializesModels
```

**Constructor parameters**:
- `string $tenantId`
- `string $title`
- `string $body`
- `string $eventType` — e.g. `invoice.paid`, `tally.sync.completed`
- `array $data = []` — arbitrary extra data
- `?int $targetUserId = null` — if null, notifies all tenant members

**`via(object $notifiable): array`**:
- Returns `['database', 'broadcast']`
- Database stores the notification in `notifications` table
- Broadcast sends the event to the tenant channel

**`toDatabase(object $notifiable): array`**:
```php
return [
    'tenant_id'  => $this->tenantId,
    'title'      => $this->title,
    'body'       => $this->body,
    'event_type' => $this->eventType,
    'data'       => $this->data,
];
```

**`toBroadcast(object $notifiable): BroadcastMessage`**:
- Creates a `BroadcastSystemNotification` event payload
- Returns `new BroadcastMessage([...])` OR explicitly fire `BroadcastSystemNotification` inside `toArray()`

**Web Push handling**: Web push is NOT done via a notification `via()` channel. Instead, after the notification is stored in DB, dispatch a queued job `SendWebPushNotification` that iterates the user's `pushSubscriptions` and sends each one via `minishlink/web-push`. This avoids coupling the notification class to the web push library.

**Alternatively** (cleaner): create a custom notification channel class `App\Notifications\Channels\WebPushChannel` and add `'web_push'` to `via()`. This is the standard Laravel pattern.

**`WebPushChannel`** (`app/Notifications/Channels/WebPushChannel.php`):
```php
public function send(object $notifiable, Notification $notification): void
{
    $subscriptions = $notifiable->pushSubscriptions;
    if ($subscriptions->isEmpty()) return;

    $webPush = new WebPush([
        'VAPID' => [
            'subject'    => config('services.vapid.subject'),
            'publicKey'  => config('services.vapid.public_key'),
            'privateKey' => config('services.vapid.private_key'),
        ],
    ]);

    $payload = json_encode($notification->toWebPush($notifiable));

    foreach ($subscriptions as $sub) {
        $webPush->queueNotification(
            Subscription::create([
                'endpoint' => $sub->endpoint,
                'keys'     => ['p256dh' => $sub->public_key, 'auth' => $sub->auth_token],
            ]),
            $payload
        );
    }

    foreach ($webPush->flush() as $report) {
        if ($report->isSubscriptionExpired()) {
            PushSubscription::where('endpoint', $report->getEndpoint())->delete();
        }
    }
}
```

Add `toWebPush(object $notifiable): array` method to `SystemNotification`:
```php
public function toWebPush($notifiable): array
{
    return [
        'title' => $this->title,
        'body'  => $this->body,
        'data'  => ['event_type' => $this->eventType, 'url' => '...'],
    ];
}
```

Add `'web_push' => \App\Notifications\Channels\WebPushChannel::class` to `config/app.php` under `notification_channels`, or register in `AppServiceProvider::boot()` via `Notification::extend('web_push', fn($app) => new WebPushChannel())`.

### 7.4 Browser Push Subscription Flow

Create a composable `resources/js/composables/usePushNotifications.js`:

```js
export function usePushNotifications() {
    const publicKey = import.meta.env.VITE_VAPID_PUBLIC_KEY;

    async function subscribe() {
        if (!('serviceWorker' in navigator) || !('PushManager' in window)) return;

        const permission = await Notification.requestPermission();
        if (permission !== 'granted') return;

        const registration = await navigator.serviceWorker.ready;
        const subscription = await registration.pushManager.subscribe({
            userVisibleOnly: true,
            applicationServerKey: urlBase64ToUint8Array(publicKey),
        });

        await axios.post(route('push.subscribe'), {
            endpoint: subscription.endpoint,
            keys: {
                p256dh: btoa(String.fromCharCode(...new Uint8Array(subscription.getKey('p256dh')))),
                auth:   btoa(String.fromCharCode(...new Uint8Array(subscription.getKey('auth')))),
            },
        });
    }

    return { subscribe };
}
```

**Service Worker**: Create `public/sw.js`:
```js
self.addEventListener('push', (event) => {
    const data = event.data.json();
    event.waitUntil(
        self.registration.showNotification(data.title, {
            body: data.body,
            icon: '/favicon.ico',
            data: data.data,
        })
    );
});

self.addEventListener('notificationclick', (event) => {
    event.notification.close();
    const url = event.notification.data?.url ?? '/';
    event.waitUntil(clients.openWindow(url));
});
```

Register the service worker in `app.js` (not `bootstrap.js`, to keep separation):
```js

}
```

## Phase 9 — File Attachments

### 9.1 Storage Disk

Use the existing `local` disk (private, served by Laravel, not publicly accessible). Path is `storage/app/private/`.

**Path pattern**: `chat/{tenant_id}/{Y_m}/{random_uuid}.{extension}`

Example: `chat/018e1234-abcd-7000-0000-000000000001/2026_05/018f9999-beef-7100-0000-000000000042.pdf`

The `{Y_m}` segment (e.g. `2026_05`) allows easy archival of old files by month.

### 9.2 Serving Files

Files are served via the signed download route `GET /t/{tenant}/groups/{room}/attachments/{attachment}/download`. The controller calls `Storage::disk('local')->download(...)` which streams the file with correct headers. This ensures:
- Authentication and membership are checked before serving
- Files are never publicly guessable by URL
- No need for S3 or signed URL infrastructure

### 9.3 Size Limits

- **Single file max**: 20 MB (`max:20480` in validation)
- **Per-message attachment count**: max 5 files (validate `attachment_ids` array has max 5 items)
- **Allowed MIME types**: `image/jpeg`, `image/png`, `image/gif`, `image/webp`, `application/pdf`, `application/msword`, `application/vnd.openxmlformats-officedocument.wordprocessingml.document`, `application/vnd.ms-excel`, `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`, `text/plain`, `text/csv`

### 9.4 MIME Type Validation

**Do NOT trust the client-supplied Content-Type header.** Use Laravel's `mimes:` validation rule which reads the actual file magic bytes:

```php
$request->validate([
    'file' => [
        'required',
        'file',
        'max:20480',
        'mimes:jpg,jpeg,png,gif,webp,pdf,doc,docx,xls,xlsx,txt,csv',
    ],
]);
```

The `mimes` rule uses `finfo_file` internally to check the actual file signature.

### 9.5 Orphan Attachment Cleanup

Attachments uploaded but never linked to a message (user abandons the chat input) become orphans. Create a scheduled job:

**File**: `app/Console/Commands/CleanOrphanChatAttachments.php`

```
Artisan command: chat:clean-orphan-attachments
Schedule: daily (in routes/console.php: Schedule::command('chat:clean-orphan-attachments')->daily())
Logic: Delete ChatAttachment records where chat_message_id IS NULL and created_at < now() - 1 hour. Also delete the actual file from storage.
```

### 10.2 Nginx WebSocket Proxy Configuration

Reverb runs on `127.0.0.1:8080`. Nginx must proxy WebSocket upgrade requests. Add to the server block for `yourdomain.com`:

```nginx
# WebSocket — Laravel Reverb
location /app {
    proxy_pass             http://127.0.0.1:8080;
    proxy_http_version     1.1;
    proxy_set_header       Upgrade $http_upgrade;
    proxy_set_header       Connection "upgrade";
    proxy_set_header       Host $host;
    proxy_set_header       X-Real-IP $remote_addr;
    proxy_set_header       X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header       X-Forwarded-Proto $scheme;
    proxy_read_timeout     60s;
    proxy_send_timeout     60s;
    proxy_connect_timeout  5s;
}

# Broadcasting auth endpoint (standard HTTP, handled by PHP-FPM)
location /broadcasting/auth {
    try_files $uri $uri/ /index.php?$query_string;
}
```

**SSL note**: In production, set `REVERB_SCHEME=https` and configure Reverb's TLS settings in `config/reverb.php` with your certificate paths, OR (preferred) let nginx handle TLS termination and keep Reverb on HTTP internally (`REVERB_SCHEME=http` with the proxy).

## Key Architectural Decisions and Rationale

**Why PresenceChannel for rooms**: Presence channels give the online users list (`here/joining/leaving`) for free, which is required for typing indicators and online status. The cost is that channel auth must return a user object — this is handled in `channels.php`.

**Why not merge with existing AI chat**: The existing `ChatController` is stateless HTTP request/response with no DB storage for messages. The new group chat is fully persistent, real-time, and architecturally separate. They share only the `AuthenticatedLayout` and permission middleware pattern.

**Why `database` queue for broadcasts**: The app explicitly uses the database queue driver. Reverb broadcasts travel through this queue as `BroadcastEvent` jobs. The existing `queue:work` process handles them. This is slightly higher latency than Redis (~50–200ms) but acceptable for chat; typing indicators use `ShouldBroadcastNow` to bypass the queue entirely.

**Why message_reads is a high-water mark table (not per-message)**: Per-message read rows would generate one DB write per user per message — at scale this is a write amplification problem. The high-water mark pattern (one row per user per room, upserted on each read) is exactly what WhatsApp and Slack use.

**Why attachments are uploaded separately before message send**: This enables a two-step flow where the file is stored first, a preview is shown, and then the message is sent with the attachment IDs. This prevents lost uploads if the message send fails, and allows the UI to show upload progress without blocking the message composer.
